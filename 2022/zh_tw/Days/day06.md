## DevOps - 真實故事

一開始，DevOps 對我們很多人來說似乎遙不可及，因為我們沒有像 Netflix 或財富 500 強公司那樣實踐它，但我認為現在它開始轉向正常，因為企業開始採用 DevOps 實踐。

你會從下面的參考資料中看到，有許多不同的行業和垂直領域使用 DevOps，因此對他們的業務目標產生了巨大的積極影響。

這裡的總體好處是，如果正確完成，DevOps 應該有助於提高你企業的軟體開發速度和質量。

我想利用這一天來看看採用 DevOps 實踐的成功公司，並分享一些相關資源。這將是社群深入參與並提供幫助的絕佳機會。你是否在企業中採用了 DevOps 文化？它是否成功？

我在上面提到了 Netflix，並將再次談到它們，因為這是一個非常好的模型，與我們今天通常看到的相比相當先進，但我還會提到其他一些在這方面取得成功的大品牌。

## Amazon

2010 年，Amazon 將他們的物理服務器佔用空間遷移到 AWS（Amazon Web Services）雲。這使他們能夠通過非常小的增量擴展容量來節省資源。我們也知道 AWS 在運行 Amazon 零售分支的同時繼續產生高收入。

Amazon 在 2011 年（根據下面的鏈接）採用了持續部署流程，他們的開發人員可以隨時部署代碼到他們需要的任何服務器。這使 Amazon 能夠實現平均 11.6 秒將新軟體部署到生產服務器！

## Netflix

誰不使用 Netflix？這是一個高質量的流媒體服務，就個人而言，具有出色的用戶體驗。

為什麼用戶體驗如此出色？好吧，提供一個沒有個人記憶故障的服務的能力需要速度、靈活性和對質量的關注。

Netflix 開發人員可以自動將代碼片段構建為可部署的 Web 鏡像，而無需依賴 IT 運營。隨著鏡像的更新，它們使用自定義構建的基於 Web 的平台集成到 Netflix 的基礎設施中。

持續監控已到位，因此如果鏡像部署失敗，新鏡像將回滾，流量將重新路由回以前的版本。

## Etsy

與我們中的許多人一樣，與許多公司一樣，圍繞緩慢和痛苦的部署確實存在鬥爭。同樣，我們可能也經歷過在有很多孤島和團隊不能很好地協作的公司在工作。

從我對 Amazon 和 Netflix 的閱讀來看，Etsy 可能在 2009 年底左右採用了讓開發人員部署他們的代碼，這可能甚至早於其他兩個。（有趣！）

我在這裡讀到的一個有趣的收穫是，他們意識到當開發人員對部署感到負責時，他們也會對應用程序性能、正常運行時間和其他目標負責。

學習文化是 DevOps 的關鍵部分。如果吸取了教訓，即使失敗也可能是成功。（不確定這個引用來自哪裡，但它有點道理！）

### 我們前幾天查看 DevOps 的回顧

- DevOps 是開發和運營的結合，允許單個團隊管理整個應用程序開發生命週期，包括**開發**、**測試**、**部署**、**運營**。

- DevOps 的主要重點和目標是縮短開發生命週期，同時與業務目標密切一致地頻繁交付功能、修復和功能。

- DevOps 是一種軟體開發方法，通過這種方法可以可靠快速地交付和開發軟體。你可能還會看到這被稱為**持續開發、測試、部署、監控**
